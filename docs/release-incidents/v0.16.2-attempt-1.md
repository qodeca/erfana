# Release incident: v0.16.2 attempt #1

- **Run URL:** https://github.com/qodeca/erfana/actions/runs/27716374257
- **Timestamp:** 2026-06-17T20:13Z
- **Failed leg(s):** Windows
- **Failed step:** Decode Azure signing certificate to PFX file
- **Signature:** `Decoded file is not a valid PKCS#12 envelope or password is wrong` (same as v0.16.1, cookbook Row 11) — but a **different root cause**.

## Root cause (operator-diagnosed): legacy-encrypted PFX rejected by OpenSSL 3.x

After v0.16.1, the `AZURE_CLIENT_CERTIFICATE_BASE64` + `AZURE_CLIENT_CERTIFICATE_PASSWORD` secrets were re-uploaded as a verified-matching pair from local material (`openssl pkcs12 -info` printed `MAC verified OK` locally with LibreSSL 3.3.6). v0.16.2 still failed at the same decode guard.

The decode step's sanity check (`.github/workflows/build_win.yml`) runs `openssl pkcs12 -info -in "$CERT_PATH" -noout -password "pass:$AZURE_CLIENT_CERTIFICATE_PASSWORD"` on the **windows-latest** runner, whose `openssl` is **OpenSSL 3.x**. The signing PFX (regenerated during the open-source setup) was exported with **legacy PKCS#12 algorithms**:

```
PKCS7 Encrypted data: pbeWithSHA1And40BitRC2-CBC, Iteration 2048
Shrouded Keybag:      pbeWithSHA1And3-KeyTripleDES-CBC, Iteration 2048
```

OpenSSL 3.0+ disables these legacy algorithms by default and cannot load such a PFX without `-legacy`. LibreSSL (macOS `/usr/bin/openssl`) still reads them, which is why local verification passed but CI failed. Critically, `@azure/identity` (Node 22 / OpenSSL 3.x) that actually consumes the PFX downstream would reject it too — so the decode guard correctly caught a real incompatibility, not a false positive.

**Verification of diagnosis (local, OpenSSL 3.6.2):**
- Legacy PFX → `openssl pkcs12 -info` **fails** without `-legacy`.
- PFX re-exported with `-keypbe AES-256-CBC -certpbe AES-256-CBC -macalg sha256` (PBES2/PBKDF2) → **loads cleanly** under OpenSSL 3.x with no `-legacy`.

## Fix applied

1. Re-exported the PFX from the unchanged private key + public cert (`azure-private.key` + `azure-public.crt`) with **AES-256 (PBES2)** encryption and SHA-256 MAC. Same cert identity → still trusted by the Azure app registration.
2. Verified it loads under OpenSSL 3.x exactly as the CI guard does.
3. Re-uploaded `AZURE_CLIENT_CERTIFICATE_BASE64` from the modern PFX (password unchanged and still correct).
4. Made the modern PFX the canonical local copy; kept the legacy one as `azure-signing-legacy-rc2.pfx` for reference; appended a runbook note.
5. v0.16.2 tag burned → re-cut as **v0.16.3**.

## Hardening opportunity (follow-up, not blocking)

The decode guard catches legacy PFXs only implicitly (as a generic parse failure). Consider either (a) documenting the AES-export requirement in `docs/build/release.md § B.3 PFX hygiene`, or (b) making the guard print the PFX algorithm / hint at `-legacy` so the next operator diagnoses it in seconds rather than via OpenSSL-version forensics. Cookbook Row 11's regex already matches this signature; its remediation text should be extended to mention the OpenSSL-3.x-legacy cause alongside the secret-mismatch cause.

See also: `docs/build/release.md` § Failure recovery, § B.3 PFX hygiene; `docs/release-incidents/v0.16.1-attempt-1.md`.
