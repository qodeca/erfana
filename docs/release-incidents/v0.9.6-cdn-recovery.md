# Release incident: v0.9.6 — Phase 4.2 CDN download recovery

- **Date:** 2026-05-22
- **Version:** v0.9.6 — **not** burned (incident was operator-side, not `release.yml`)
- **Phase:** 4.2 (download of all 11 release assets in the local skill's verify step)
- **release.yml run:** [26280699390](https://github.com/qodeca/erfana/actions/runs/26280699390) — `completed/success` (all 6 jobs green in 30 min)
- **Outcome:** release published as latest after recovery; gates 4.3–4.5 verified cleanly

## Symptom

After `release.yml` finished green, the operator's Phase 4 verify script ran `gh release download v0.9.6 --pattern '*' --clobber --dir $WORK` and progressed for ~17 minutes, completing 7 of 11 assets (Windows `.exe`s, macOS `.dmg` + `.zip`, `SHA256SUMS` + `.minisig`). With 4 assets still in flight at 49–71%, `gh` exited 1 with:

```
read tcp 192.168.10.148:59603->185.199.111.133:443: read: operation timed out
```

`185.199.111.133` resolves to Fastly's edge — the CDN GitHub uses for release-asset downloads. The TCP connection was idle long enough for one side to time out. `gh release download` has no built-in retry on this class of error in `gh` 2.91.0.

## Recovery

A small bash script ran 4 parallel `curl -C -` against the 4 partial files. Key flags:

```bash
curl -fL -C -                              \
  --retry 10 --retry-delay 5               \
  --retry-max-time 1800                    \
  --retry-connrefused --retry-all-errors   \
  --connect-timeout 30                     \
  -H "Accept: application/octet-stream"    \
  -H "Authorization: Bearer $(gh auth token)" \
  -H "X-GitHub-Api-Version: 2022-11-28"    \
  -o "$file"                               \
  "$api_url"
```

Where `$api_url` is the per-asset `.assets[].apiUrl` from the JSON the skill cached as `release-meta.json` (e.g. `https://api.github.com/repos/qodeca/erfana/releases/assets/426978320`).

Resume worked on first attempt; all 4 streams sustained ~1.5 MB/s each (~6 MB/s aggregate vs `gh release download`'s observed ~0.75 MB/s in the same network conditions). Total recovery for the remaining ~500 MB: ~3 minutes.

## Why this works

- The asset `apiUrl` 302-redirects to a signed S3 URL. S3 supports HTTP `Range` requests natively, which is what `curl -C -` sends.
- A fresh redirect target is generated per `curl` invocation, sidestepping the original idle-killed connection.
- Parallel-per-file (one `curl` per asset) avoids serializing on the slowest stream.

## Why `gh release download --clobber` is not the answer

`--clobber` overwrites the existing file — each retry starts from byte 0. With 7 of 11 already complete at the time of failure, a re-run would needlessly re-download ~1.5 GB. The curl resume preserves the existing bytes.

`gh release download` does not currently expose a `--resume` flag (as of `gh` 2.91.0).

## What about the cryptographic gates?

All three gates passed cleanly once the downloads completed:

- **4.3 minisign** — `Signature and comment signature verified` under the PRIMARY key (timestamp `1779444902`).
- **4.4 per-asset SHA-256** — 9 of 9 binaries match the signed `SHA256SUMS`.
- **4.5 CI-recorded digest** — draft `SHA256SUMS` byte-equal to the `sha256sums-digest` workflow artifact.

This confirms the partial-then-resumed bytes are byte-identical to what the CI signed. Resume does not weaken the trust chain.

## Tag-burn rule does not apply

The skill's Rule #6 ("verification failure aborts; the release is burned; bump the patch") applies to cryptographic failures in gates 4.3–4.5. Phase 4.2 is a transport-layer step on the operator's side; the tag, the draft release, and all signed artifacts on GitHub were untouched throughout the incident.

## Suggested permanent mitigation

`docs/build/release.md § Failure recovery` now documents this row so future operators don't re-discover the pattern. Further improvements to consider:

- Wrap the Phase 4 verify script's `gh release download` in a retry loop with this `curl -C -` fallback baked in (so the skill self-recovers without a one-off script).
- Upstream a `--resume` flag to `gh release download` (would also help non-Erfana users).
