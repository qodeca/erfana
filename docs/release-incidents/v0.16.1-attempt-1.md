> `Decoded file is not a valid PKCS#12 envelope or password is wrong`

Proposed new cookbook row regex: `Decoded file is not a valid PKCS#12 envelope or password is wrong`

**Root cause (operator-diagnosed):** `AZURE_CLIENT_CERTIFICATE_BASE64` and
`AZURE_CLIENT_CERTIFICATE_PASSWORD` GitHub Secrets are mismatched — either the certificate was
regenerated (new PFX exported with a new password) and only one of the two secrets was updated,
or the base64 payload itself is corrupted (wrong source file encoded). The `openssl pkcs12 -info`
sanity check catches this early (before `@azure/identity` inverts the failure 20+ min into the
build), confirming the decode produced a valid-size but non-loadable PFX. The size gate passed
(`SIZE >= 100`), so the binary is non-empty — ruling out a missing or fully-blank secret. This is
a content/password mismatch, not a whitespace/encoding issue (which Row 9 covers for
`AZURE_SIGNING_ENDPOINT`).

## Suggested next action

This is an unmatched signature — no cookbook row covers PKCS#12 password mismatch. Operator
steps:

1. **Locate the signing-secrets material** on the signing workstation (the machine where the
   PFX was originally generated). This is likely in the key vault described in
   `docs/build/release.md § B.2 cert workstation-loss DR` and `§ B.3 PFX hygiene`.

2. **Determine which secret is stale.** Two scenarios:
   - *Scenario A — cert was regenerated, password updated, but base64 not re-uploaded:*
     re-export the PFX from the keystore and re-encode:
     ```bash
     openssl base64 -in signing.pfx | tr -d '\n' | pbcopy
     # Then: gh secret set AZURE_CLIENT_CERTIFICATE_BASE64 --repo qodeca/erfana
     # (paste from clipboard, or pipe: printf '%s' "$(pbpaste)" | gh secret set ...)
     ```
   - *Scenario B — cert was regenerated with a new password, old password left in secret:*
     update `AZURE_CLIENT_CERTIFICATE_PASSWORD` to match the password used when the PFX was
     exported. Use `printf '%s' '<password>' | gh secret set AZURE_CLIENT_CERTIFICATE_PASSWORD`
     (no trailing newline — see Row 9 defensive notes).
   - *Scenario C — base64 encoding is from the wrong file (e.g., a CA cert or CSR rather
     than the PFX):* verify locally that the base64 decodes to a valid PKCS#12 envelope:
     ```bash
     printf '%s' "$THE_BASE64_VALUE" | tr -d '\r\n' | openssl base64 -d -A > /tmp/test.pfx
     openssl pkcs12 -info -in /tmp/test.pfx -noout -passin pass:'<password>'
     # Should print: MAC verified OK
     ```

3. **Update both secrets atomically** — whenever the PFX changes, always update
   `AZURE_CLIENT_CERTIFICATE_BASE64` AND `AZURE_CLIENT_CERTIFICATE_PASSWORD` together to
   prevent the mismatch that caused this failure.

   using the "NEW SIGNATURE" details above so the next occurrence resolves automatically.
