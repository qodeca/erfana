# Source grounding – specification gap analysis

> Part of [Source grounding research](README.md)
> Related: [Missing specs and recommendations](missing-specs.md) | [Implementation roadmap](implementation-roadmap.md)

## 1. Executive summary

This document identifies gaps between the current erfana specification and what is required for a production-quality source grounding system. The analysis covers retrieval, ranking, citation, verification, and cross-cutting technical concerns.

- **Gaps identified**: 15 (including Gap 0)
- **Missing specifications**: 8
- **Cross-cutting concerns**: 15

Each gap includes what is currently specified, what is missing, the impact of the gap, and a suggested amendment. Gaps are numbered 0–14; missing specifications are lettered A–H.

## 2. Gap inventory

### 2.0 Foundational gap

#### Gap 0: Source/derived document distinction

| Aspect | Detail |
|--------|--------|
| **What's specified** | All markdown files in the vault are treated identically by the indexing and retrieval pipeline. No metadata distinguishes authored source material from generated or derived documents. |
| **What's missing** | No formal distinction between **source** (ground truth – human-authored, externally referenced, or imported primary material) and **derived** (generated by LLM, computed summaries, synthesised reports) documents. |
| **Impact** | The system could cite a derived document as evidence for a claim, creating **circular reasoning** – the LLM cites its own previous output as factual grounding. Confidence ratings become meaningless when the provenance of cited material is unknown. Users cannot trust grounding scores if derived content is mixed with authoritative sources. |
| **Suggested amendment** | 1. **Frontmatter tag** – require `type: source` or `type: derived` in document frontmatter. Default to `derived` for safety (opt-in for source status). 2. **Folder convention** – recommend `sources/` and `derived/` top-level directories, with the tag taking precedence when present. 3. **Source registry** – maintain a `sources.json` manifest listing all verified source documents with hash, import date, and origin URL. 4. **MCP `sources_only` parameter** – add a boolean parameter to search tools that restricts results to `type: source` documents only. Default to `true` for grounding queries. 5. **UI distinction** – render source and derived citations differently (e.g. solid vs dashed citation markers, distinct colours in the grounding panel). |

> **Note**: This must be the first specification created – all other grounding features depend on the ability to distinguish source from derived material.

### 2.1 Retrieval gaps

#### Gap 1: Section-level retrieval granularity

| Aspect | Detail |
|--------|--------|
| **What's specified** | Full-text search returns document-level results via SQLite FTS5. The MCP `search_notes` tool returns `title`, `path`, `snippet`, and `score`. |
| **What's missing** | No section-level or paragraph-level retrieval. When a 5,000-word document matches a query, the entire document is returned with a short snippet – the LLM must scan the full text to find the relevant passage. |
| **Impact** | Increased token consumption, slower response times, and higher risk of the LLM hallucinating when the relevant section is buried in a long document. Precision of citation drops because the system cannot point to a specific section. |
| **Suggested amendment** | Index documents at the heading-section level (H1–H3). Each section becomes a retrievable unit with its own embedding vector, FTS entry, and metadata (parent document, heading hierarchy, character offset). Return the matched section plus one level of surrounding context. See **GINGER** (SIGIR 2025, [arXiv:2503.18174](https://arxiv.org/abs/2503.18174)) – uses atomic 'information nuggets' rather than section-level retrieval, achieving SOTA on TREC RAG'24. |

#### Gap 2: No vector/semantic search

| Aspect | Detail |
|--------|--------|
| **What's specified** | Keyword-based search via FTS5 with BM25 ranking. The `search_notes` tool accepts a text query and returns lexical matches. |
| **What's missing** | No semantic similarity search. Queries like "What are the risks of the current approach?" will fail if the source uses different terminology (e.g. "challenges", "concerns", "limitations"). |
| **Impact** | Low recall for conceptual queries. Users must guess the exact wording used in their sources. The grounding system misses semantically relevant passages that use synonyms or paraphrases. |
| **Suggested amendment** | Add a vector search path using local embeddings (ONNX-based model, e.g. `all-MiniLM-L6-v2` for initial release, upgrade path to `nomic-embed-text-v1.5` or `gte-small`). Store embeddings in `sqlite-vec`. Implement hybrid search combining BM25 + cosine similarity with reciprocal rank fusion (RRF). |

#### Gap 3: No `full_text` field in search results

| Aspect | Detail |
|--------|--------|
| **What's specified** | Search results include `snippet` (truncated match context) but not the full section or document text. |
| **What's missing** | The LLM cannot read the full matched content without a separate `read_note` call. This adds latency and requires the LLM to decide which results warrant a full read. |
| **Impact** | Two-step retrieval increases response time and token usage. The LLM may skip reading a relevant result due to tool call budget, leading to incomplete grounding. |
| **Suggested amendment** | Add an optional `include_full_text: boolean` parameter to `search_notes`. When true, return the full section text (for section-level retrieval) or full document text (for document-level). Default to `false` to preserve backward compatibility. Cap at a configurable token limit (e.g. 8,000 tokens per result). |

#### Gap 4: No heading-level anchoring

| Aspect | Detail |
|--------|--------|
| **What's specified** | Citations reference documents by file path and title. |
| **What's missing** | No heading slug or anchor system. Citations cannot point to a specific section within a document. |
| **Impact** | Users must manually search within a cited document to find the relevant passage. For long documents, this degrades the citation experience significantly. |
| **Suggested amendment** | Generate stable heading slugs during indexing (GitHub-compatible slug algorithm). Include `heading_slug` and `heading_text` in search results. Construct deep links as `path#heading-slug` for UI navigation. |

### 2.2 Ranking and relevance gaps

#### Gap 5: No chunk/section metadata for ranking

| Aspect | Detail |
|--------|--------|
| **What's specified** | BM25 score is the sole ranking signal. |
| **What's missing** | No metadata-based ranking signals: document freshness (modification date), section depth (H1 vs H3), document type (source vs derived), link density, or citation frequency. |
| **Impact** | All documents are ranked equally regardless of recency, authority, or structural importance. A stale derived note ranks the same as a recently updated source document. |
| **Suggested amendment** | Implement a composite scoring function: `final_score = w1 * bm25 + w2 * semantic_similarity + w3 * freshness_decay + w4 * source_type_boost`. Expose weights as configuration. Boost `type: source` documents by default. See **Contextual Retrieval** (Anthropic, 2024) – adding context to chunks before embedding improves retrieval by 49%. Also **Late Chunking** (JinaAI, 2024) – embed full documents first, then chunk embeddings to preserve cross-chunk context. |

#### Gap 6: No `matched_terms` in results

| Aspect | Detail |
|--------|--------|
| **What's specified** | Search returns a score but not which terms contributed to the match. |
| **What's missing** | No indication of which query terms matched, their positions, or their contribution to the score. |
| **Impact** | The LLM cannot assess match quality or explain why a result was returned. UI cannot highlight matching terms accurately. |
| **Suggested amendment** | Return `matched_terms: string[]` and optionally `term_positions: {term, offset, length}[]` in search results. Use FTS5 `highlight()` and `snippet()` functions with custom markers. |

#### Gap 7: No topic or tag filtering

| Aspect | Detail |
|--------|--------|
| **What's specified** | Search accepts a free-text query. The `list_notes` tool supports `folder` filtering. |
| **What's missing** | No filtering by topic, tag, frontmatter field, or document type. Users cannot restrict grounding to a specific domain (e.g. "only search architecture documents" or "only source documents"). |
| **Impact** | Cross-domain contamination – a query about "performance" might return results about application performance, team performance reviews, and financial performance. The `sources_only` filter (Gap 0) is a special case of this broader need. |
| **Suggested amendment** | Add `topic_filter`, `tag_filter`, and `type_filter` parameters to search tools. Index frontmatter tags and topics during import. Support AND/OR filter logic. |

### 2.3 Extraction and structuring gaps

#### Gap 8: No structured entity/claim extraction

| Aspect | Detail |
|--------|--------|
| **What's specified** | Documents are stored and searched as plain text. No structured extraction occurs during import. |
| **What's missing** | No extraction of named entities (people, organisations, dates, technical terms), key claims, definitions, or relationships from source documents. |
| **Impact** | The system cannot answer entity-centric queries efficiently ("Who authored the architecture decision?", "When was the API deprecated?"). Relationship-based navigation is impossible. |
| **Suggested amendment** | Implement LLM-based extraction via Claude Code as a post-import prompt template. After a document is imported or updated, run a configurable extraction prompt that identifies entities, claims, definitions, and relationships. Store extracted triples in a structured table linked to the source section. This avoids the complexity of NLP pipelines while leveraging the LLM's understanding. See **AGREE** (NAACL 2024, [arXiv:2311.09533](https://arxiv.org/abs/2311.09533)) – adaptive grounding with retrieval-augmented generation for multi-step entity and claim extraction. |

#### Gap 9: No transcript/meeting note structuring

| Aspect | Detail |
|--------|--------|
| **What's specified** | All markdown files are indexed uniformly. No special handling for transcripts, meeting notes, or other semi-structured formats. |
| **What's missing** | Transcripts and meeting notes have unique structural patterns (speaker turns, timestamps, action items, decisions) that are lost when treated as plain prose. |
| **Impact** | Key decisions and action items are buried in transcript noise. The system cannot distinguish a decision from a casual remark. Speaker attribution is lost. |
| **Suggested amendment** | Add a transcript structuring pipeline: detect transcript format, extract speaker turns, identify decisions/action items, and generate a structured summary. Store both the raw transcript and the structured version. |

> **Dependency note**: This gap is premature without transcript structuring (Missing Spec C) implemented first. The structuring pipeline must exist before transcript-specific retrieval can be designed.

### 2.4 Verification and confidence gaps

#### Gap 10: No claim-level verification

| Aspect | Detail |
|--------|--------|
| **What's specified** | The grounding system retrieves relevant passages but does not verify whether the LLM's generated claims are supported by the retrieved evidence. |
| **What's missing** | No mechanism to compare generated statements against source passages and flag unsupported, contradicted, or partially supported claims. |
| **Impact** | The LLM can generate plausible-sounding claims that are not grounded in any source. Users see citations but have no assurance that the cited passage actually supports the claim. This is the core grounding failure mode. |
| **Suggested amendment** | Implement a post-generation verification step: for each claim in the response, check whether at least one retrieved passage entails the claim. Use NLI-style classification (entailed / neutral / contradicted). Flag contradicted claims with a warning. Display confidence indicators per claim, not just per response. |

#### Gap 11: No grounding confidence scoring

| Aspect | Detail |
|--------|--------|
| **What's specified** | Search results have a BM25 relevance score. No grounding-specific confidence metric exists. |
| **What's missing** | No composite confidence score that considers: retrieval relevance, source authority (source vs derived), passage-claim alignment, temporal freshness, and corroboration (multiple sources agreeing). |
| **Impact** | Users cannot distinguish between well-grounded responses (multiple authoritative sources, high alignment) and weakly grounded ones (single tangential match). |
| **Suggested amendment** | Define a `grounding_confidence` metric with documented components and thresholds. Display as a visual indicator (e.g. high/medium/low or a percentage). Require minimum confidence before presenting a response as "grounded". See **FACTS Grounding** (Google DeepMind, 2025, [arXiv:2501.03200](https://arxiv.org/abs/2501.03200)) – benchmark for evaluating factual grounding of LLM outputs against source documents, with automated evaluation methodology. |

### 2.5 Citation and output gaps

#### Gap 12: No citation format specification

| Aspect | Detail |
|--------|--------|
| **What's specified** | The MCP tools return file paths and titles. No standard citation format is defined for LLM responses. |
| **What's missing** | No specification for how citations should appear in generated text. Should they be inline `[1]` markers? Footnotes? Hyperlinks? How should multiple citations be grouped? |
| **Impact** | Inconsistent citation formatting across responses. The UI cannot reliably parse and render citations. Users cannot click through to sources. |
| **Suggested amendment** | Define a citation format standard: inline markers `[n]` with a references section at the end. Each reference includes: document title, section heading (if available), file path, and confidence indicator. Define a machine-parseable format (JSON) alongside the human-readable format. |

#### Gap 13: No grounding audit trail

| Aspect | Detail |
|--------|--------|
| **What's specified** | No logging of grounding operations. |
| **What's missing** | No record of which queries were made, which documents were retrieved, which passages were cited, and what confidence scores were assigned. |
| **Impact** | Cannot debug grounding failures ("why did the system cite this irrelevant document?"). Cannot audit grounding quality over time. Cannot identify systematic gaps in source coverage. |
| **Suggested amendment** | Log grounding operations to a structured table: query, timestamp, retrieved documents (with scores), cited passages, confidence scores, and user feedback (if any). Provide a query audit log view in the UI. See **FACTS Grounding** (Google DeepMind, 2025, [arXiv:2501.03200](https://arxiv.org/abs/2501.03200)) – provides evaluation methodology applicable to audit trail design. |

#### Gap 14: Whitespace normalisation contradiction

| Aspect | Detail |
|--------|--------|
| **What's specified** | The specification mentions both preserving original whitespace for faithful rendering and normalising whitespace for consistent search indexing. |
| **What's missing** | No resolution of the contradiction. It is unclear whether the system should store normalised or original text, or maintain both representations. |
| **Impact** | Implementation ambiguity. If normalised text is stored, code block formatting and table alignment break. If original text is stored, search may miss matches due to inconsistent whitespace. |
| **Suggested amendment** | Store original text for display; create a normalised index for search. Document the normalisation rules (collapse runs of whitespace, trim lines, normalise line endings to `\n`). Ensure `snippet` results show original formatting while search operates on the normalised index. |
