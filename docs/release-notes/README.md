# Release notes

This directory holds per-version release-notes markdown for Erfana, one file per published version. They serve both GitHub release bodies (via `gh release create --notes-file`) and in-repo archaeology.

## Naming convention

```
docs/release-notes/v{major}.{minor}.{patch}.md
```

Strict semver only — pre-release suffixes (`-rc`, `-beta`) are out of scope for the current release pipeline (see [`docs/build/release.md`](../build/release.md) §Scope). One file per version. Once a file is written and tagged, treat its contents as immutable — bugs in the notes get fixed in a follow-up version, not by editing the archived file.

## Two-tier format

Every release notes file has two sections:

1. **User-facing summary** — 3–5 bullet points describing what changed from an end user's perspective. Operator-curated during the release skill run. Plain language, no commit SHAs, no issue numbers, no dev-internal jargon.
2. **Technical changes** — generated by [`git cliff`](https://git-cliff.org/) from commit history between the previous tag and the current tag. Wrapped in a `<details>` block so the summary dominates by default and the technical list is one click away for those who want it.

Structure:

```markdown
# Erfana v{VERSION}

_Released: {YYYY-MM-DD}_

- summary bullet one
- summary bullet two
- summary bullet three

<details>
<summary>Technical changes</summary>

### Features

- feat: add dark-mode toggle
- feat(import): support .odt via LiteParse

### Bug fixes

- fix(terminal): restore scrollback on reconnect

...

</details>
```

## How files here get created

The [`releasing-erfana`](../../.claude/skills/releasing-erfana/SKILL.md) skill drives release notes generation during its Phase 1:

1. Skill runs `git cliff --tag v{VERSION} --unreleased` → produces the technical section content.
2. Skill invokes `AskUserQuestion` to collect the 3–5 bullet operator summary (subagents cannot call `AskUserQuestion`; this must happen at the skill level).
3. Skill invokes the `release-notes-drafter` agent with both inputs.
4. Agent composes the final markdown and writes it to `docs/release-notes/v{VERSION}.md`.
5. Single commit bundles this file + `package.json` version bump + `docs/CHANGELOG.md` append.

See [`docs/build/release.md § Release notes`](../build/release.md) for the full release procedure.

## Section mapping

The technical section's group headings (`Features`, `Bug fixes`, etc.) come from [`cliff.toml`](../../cliff.toml) at the repo root. Current mapping:

| Conventional commit prefix | Section |
|---|---|
| `feat` | Features |
| `fix` | Bug fixes |
| `perf` | Performance |
| `refactor` | Refactoring |
| `docs` | Documentation |
| `test` | Tests |
| `ci` | CI |
| `build` | Build |
| `chore` | Chores |
| `style` | *(skipped)* |
| everything else | *(skipped)* |

Group rendering order in the output follows the parser declaration order above, so user-relevant groups appear first.

## First release on this format

The first Erfana release using this two-tier format is **v0.9.5** — the release shipping the signed multi-platform pipeline from [#174](https://github.com/qodeca/erfana/issues/174). Earlier releases (v0.9.4 and older) predate the format; their full history is in [`docs/CHANGELOG.md`](../CHANGELOG.md).
